## 2024-07-08 - Prevent Information Leakage in API Error Responses
**Vulnerability:** Fastapi's `HTTPException` was returning the raw exception string `str(e)` on a 500 error, exposing potentially sensitive internal server details or stack traces to the end user.
**Learning:** This is a common pattern when developers quickly add a try-catch block but forget that error details returned to the client can leak sensitive architecture information, path structures, or database details.
**Prevention:** Always log the verbose error details internally (e.g., using `logger.error`), but return a generic, non-descriptive error message (e.g., "An internal error occurred") in the HTTP response.
## 2024-05-30 - [Fix SSRF vulnerability in job-outreach pipeline]
**Vulnerability:** The Python script `job-outreach/automation/run_repo.py` took external URLs from an inbox and explicitly fetched their contents via `requests.get` to scrape jobs without validating the origin or resolved IP address. This allowed a potential Server-Side Request Forgery (SSRF) attack where an attacker could provide URLs pointing to internal networks, loopback addresses, or private metadata endpoints (like `http://169.254.169.254/`).
**Learning:** External URLs supplied by users or from unverified sources (like an inbox file) should always be treated as untrusted and validated prior to being used in server-side outbound HTTP requests.
**Prevention:** Implement validation helpers (like `is_safe_url`) to check the URL scheme, resolve the hostname, and ensure the target IP is not private, loopback, or link-local. Alternatively, use a hardened proxy for external HTTP fetches.
